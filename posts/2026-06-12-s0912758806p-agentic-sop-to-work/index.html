解决的工程痛点

LLM Agent执行多步流程时存在三个高频失败模式：臆造数据、无法证明产出来源、以及agent逐渐退化为难以审计的mega-agent。团队在合规或受监管场景下（如GMP、金融审计）需要将SOP工程化但缺乏零依赖、可本地验证的编排工具。

⚙

CTO 级技术审计

项目将「Human SOP → single-tool skill → run.py 编排 → per-step gate → DRAFT → human approval」作为核心路径。编排引擎 plugins/agentic-sop-kit/kit/workflow/run.py 读取 flow.json 定义的步骤链，每步产出标准化 JSON artifact（schema/produced_by/data/trace 四字段，见 plugins/agentic-sop-kit/kit/lib/kit.py:68 的 artifact() 函数），步间通过文件交接而非内存共享。四种确定性闸门（plugins/agentic-sop-kit/kit/lib/gates.py）均为纯函数、无网络无LLM：cmd_gate 检查exit code、schema_gate 验必填字段、trace_gate 要求每个data值在trace中可溯源（gates.py:30 中对unsourced值直接block）、recompute_gate 用 math.isclose 重算聚合并比对声称值。控制流由 plugins/agentic-sop-kit/kit/lib/flow.py 的 resolve_branch 实现forward-only条件分支，predicates支持==、!=、<、>、in、exists等操作，when匹配始终优先于default（flow.py:42）。Stop-hook 回归闸门（plugins/agentic-sop-kit/hooks/stop_gate.py）在Claude每次停止时触发verify.py，失败则emit decision:block 将详情feed回Claude修复，并用持久化retry计数器防无限循环（默认上限3，stop_gate.py:58）。引擎和所有skill仅依赖Python标准库，由 tests/test_no_third_party.py 用AST静态扫描强制执行。

trace_gate 的反臆造设计（gates.py:28-37）——要求每个输出值必须在上游trace中以str匹配出现，这是一种轻量但有效的「output must be sourced」约束，比关键词匹配强。

test registry 机制（tests/registry.json）——verify.py会交叉比对flow.json中用到的skill与registry登记的测试，未登记即fail-loud（exit 3），形成「新增skill必须有测试」的编码约束。

plugins/agentic-sop-kit/kit/workflow/run.py（编排引擎核心）未在code_bundle中提供，无法审计其step执行、fix-loop重试逻辑、branch/map实现的具体代码质量，这是本次评审最大的盲区。

plugins/agentic-sop-kit/hooks/stop_gate.py 与 plugins/agentic-sop-kit/kit/hooks/stop_regression.py 代码高度重复（retry计数、block emit、上限判断逻辑几乎逐行相同），约250行中有~180行可抽取为共用模块但未做，技术债明显。

(1) 补审run.py确认fix-loop是否真能防「偷改输出过门」；(2) 将两个stop hook的共用逻辑抽取到kit/lib/；(3) trace_gate的str匹配在数值精度场景（如12.0 vs 12.00）可能有误判，建议增加normalize层；(4) 目前仅有demo级别的skill（extract/compute/report），真实业务skill的验证仍需用户自行补充。

⚠

隐藏风险与雷点

  • run.py编排引擎核心未在code_bundle中提供，fix-loop和branch/map的具体实现无法验证
  • Fork/Star比仅6.7%且仓库仅3天，社区健康度极低，单人维护风险高
  • trace_gate以str(value)做精确匹配，数值精度差异（12.0 vs 12.00）可能导致误block
  • 示例skill（extract/compute/report）仅为demo数据处理，真实业务场景的gap未被覆盖
  • summary 过长，可能含废话
◈

生态与商业价值

在Claude Code plugin生态中填补了「确定性编排+安全闸门」的空白，面向制药、金融等合规场景的团队有实际采用价值；但其商业天花板受限于Claude Code插件市场的用户规模和项目对单一人维护者的依赖。
