解决的工程痛点

AI Agent 或 CI 脚本驱动交互式终端程序时，GNU screen 的老终端模拟器不识别现代 VT 序列导致重绘失真、peek 输出不可靠；tmux 虽强但模型不同且脚本接口缺乏「屏幕语义等待」能力——sleep-poll 循环脆弱且慢。

⚙

CTO 级技术审计

boo 采用 client-daemon 架构（README 架构图），src/main.zig 中 cmdNew 通过 bindListen 创建 Unix socket 后 fork 出 daemon 子进程（src/main.zig:~runDaemon），daemon 在 src/daemon.zig 的 Daemon.loop 中以单线程 poll(2) 循环同时监听 listen fd、SIGCHLD self-pipe、所有客户端连接 fd 和 PTY master fd。PTY 分配在 src/pty.zig:spawnInPty 中完成，使用原生 C 绑定（posix_openpt/grantpt/unlockpt），子进程通过 setsid + TIOCSCTTY 获取控制终端，父进程关闭 slave 端。src/window.zig:Window.create 将 PTY master 与 ghostty-vt Terminal 和 TerminalStream 绑定，所有子进程输出经 stream.nextSlice 解析，terminal 状态包含完整屏幕、滚动历史、光标和终端模式。attach 时 repaint 方法（src/window.zig）通过 VT TerminalFormatter 从模拟器状态重新生成字节流，经 altscreen.Filter 过滤掉 alternate-screen 切换序列后发送给客户端。客户端 src/client.zig:attach 将 TTY 切到 raw mode 和 alternate screen，用 poll 三路复用（tty stdin、socket、signal pipe），信号通过 self-pipe trick 处理 SIGWINCH/SIGTERM/SIGHUP。

src/window.zig:feedDiscarded 与 src/altscreen.zig:Filter 的配合——passthrough 模式下 alt-screen 切换被过滤，被丢弃的字节仍通过 feedDiscarded 注入 terminal 模拟器，且此时 feeding_discarded=true 强制 daemon 端回答终端查询（DSR、DA 等），避免应用程序因查询无响应而挂起（src/window.zig:effectWritePty 的双条件判断）。

src/keys.zig:Parser 同时支持传统 0x01 escape 字节和 kitty keyboard CSI-u 编码的 prefix 识别，包含 press/repeat/release 事件区分、修饰键过滤（isModifierKey）、caps lock 忽略、序列跨 feed 分片和缓冲区溢出回放——约 300 行状态机有 18 个 test case 覆盖。

daemon 单线程 poll 模型下，src/daemon.zig:serviceWindow 中对每个 PTY 读取块执行 alt_filter.feed + win.feed + win.feedDiscarded，如果 child 输出极快且 alt-screen 频繁切换，repaint 会产生大量 VT 字节在同一个 poll dispatch 内串行发送，可能造成客户端积压；未见背压或流控机制。

src/daemon.zig:handleCommand 中 peek 命令将完整屏幕 dump 截断到 protocol.max_payload（1MB），但 src/protocol.zig:max_payload 是 hard limit，对于 512K 行 scrollback 场景可能丢失内容且无截断警告返回给调用方。

agent 自动化场景可直接投入使用 boo new -d -- bash + send/peek/wait 循环，但需注意 wait --text 是客户端轮询（src/main.zig:cmdWait 每 50ms 发一次 peek 控制命令），高频场景下 daemon 负载线性增长；建议在 daemon 端实现 server-side wait 语义以降低 round-trip。

隐藏风险与雷点

  • README 明确列出：一次仅一个 attached client、单窗口无 split/tab、C-a prefix 不可配置、paste 中 0x01 会被误识别为 prefix

生态与商业价值

Coder 团队出品，与 Coder 云 IDE 生态形成互补——远程开发环境中的持久会话管理是刚需。作为 tmux 的「screen 模型 + 现代 VT 引擎」替代品，在 AI Coding Agent 精确屏幕交互这一新兴垂直场景有清晰的差异化价值，但需警惕单窗口/单客户端限制对生产场景的约束。
