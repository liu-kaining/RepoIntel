解决的工程痛点

国内开发者使用 Codex CLI 时，需要手动编辑 ~/.codex/config.toml 切换模型 provider 和 base_url，且 Responses API 与 Chat Completions 格式不兼容，导致中转站/国产 API 直接对接困难。每次切换模型的成本约 5-10 分钟，且容易改错配置。

⚙

CTO 级技术审计

项目是一个 Axum HTTP 本地代理，监听 127.0.0.1:25543（src/config/mod.rs:22 DEFAULT_PORT），拦截 Codex CLI/Desktop 的 /v1/responses 和 /v1/chat/completions 请求，完成协议转换后转发到国产 LLM 的 Chat Completions API。核心链路为：src/proxy/mod.rs:246 proxy_responses → convert_responses_to_chat（src/proxy/responses_to_chat.rs:8）将 Responses API 的 input/tools/tool_choice 翻译为 Chat Completions 格式，再经 src/proxy/mod.rs:217 forward_request 转发；返回路径上，流式响应经 src/proxy/codex_compat/streaming_codex_chat.rs 的 create_responses_sse_stream_from_chat 将 Chat SSE 翻译回 Responses API 的命名事件流（response.created / response.output_text.delta / response.completed）。非流式响应走 chat_json_to_responses_json（src/proxy/mod.rs:358）做 JSON 包装。

SSE 流式协议转换层设计得相当完整。src/proxy/codex_compat/sse.rs:56 的 append_utf8_safe 正确处理了 UTF-8 多字节字符跨 chunk 边界的情况（测试覆盖了 emoji、中文、invalid byte 等场景），streaming_codex_chat.rs 内部用状态机（ChatToResponsesState）逐 chunk 解析 chat delta 并合成 Responses 事件，支持 reasoning_content / think 标签剥离 / tool_call arguments 拼接归一化，代码量虽大但逻辑路径有 10 个 #[tokio::test] 覆盖。

配置同步链路细致。src/codex/mod.rs:41 inject_proxy_config 不仅写 config.toml，还同步 auth.json、.env、Windows 用户环境变量（src/env_sync.rs:26 sync_windows_user_env 用 setx）、Desktop 会话 SQLite（src/codex/desktop_state.rs:14 sync_threads_to_config），并在写入后校验 base_url 是否正确（src/codex/mod.rs:171 verify_written_proxy_url）。自动备份 10 份 config 历史（src/codex/mod.rs:12 backup_codex_config）。

API Key 明文存储在 ~/.codex-helper/.env（src/config/mod.rs:156 save_env_value 直接 write_atomic），README 提到 keys.enc 加密存储但源码中未见加密实现——src/paths.rs 只有 helper_env_path 指向 .env 明文文件。README 与实际行为不符。

src/proxy/mod.rs:197 forward_request 中 _headers: HeaderMap 参数被忽略（下划线前缀），意味着 Codex 客户端发来的 Accept / OpenAI-Beta 等头全部丢失。虽然注释说「避免中转站报 Unsupported content type」，但这是一刀切的做法，可能导致某些中转站需要的自定义头也丢失。

建议先修正 README 中 keys.enc 的描述（或补上加密存储），然后考虑保留安全头转发（如 Authorization 的 Bearer token 已由 helper 自行注入，这部分没问题）。项目对 Codex 新版本的 Responses API 适配做得很好，但 upstream 模型名如 gpt-5.5 / gpt-5.4-mini 是虚构的（对应 custom 中转站预设），需在 README 或代码中注明这些是占位名称。

⚠

隐藏风险与雷点

  • README 声称 keys.enc 加密存储，但源码中 API Key 明文写入 .env 文件，存在安全误导
  • 项目高度耦合 Codex Desktop 的 config.toml 格式和 Responses API 协议，上游任何 breaking change 都会导致代理失效，且仓库仅 2 天历史，维护持续性未知
  • summary 过长，可能含废话
◈

生态与商业价值

解决了国内 Codex 用户的刚需痛点——直接降低使用国产模型的技术门槛。若 Codex CLI 用户量持续增长，作为其生态配套工具有较好的分发潜力；但项目高度依赖 Codex 的配置格式和 API 协议，上游变动将直接带来维护压力。
