<section class="content-panel content-panel--pain" id="pain">
<header class="panel-head">
  <span class="panel-icon" aria-hidden="true">◎</span>
  <h2 class="panel-title">解决的工程痛点</h2>
</header>
<div class="panel-body prose">
<p>AI Agent 或 CI 脚本驱动交互式终端程序时，GNU screen 的老终端模拟器不识别现代 VT 序列导致重绘失真、peek 输出不可靠；tmux 虽强但模型不同且脚本接口缺乏「屏幕语义等待」能力——sleep-poll 循环脆弱且慢。</p>
</div>
</section>

<section class="content-panel content-panel--audit" id="audit">
<header class="panel-head">
  <span class="panel-icon" aria-hidden="true">⚙</span>
  <h2 class="panel-title">CTO 级技术审计</h2>
</header>
<div class="panel-body prose">
<p class="audit-callout audit-callout--intro">boo 采用 client-daemon 架构（README 架构图），<code class="code-ref">src/main.zig</code> 中 <code class="code-ref">cmdNew</code> 通过 <code class="code-ref">bindListen</code> 创建 Unix socket 后 fork 出 daemon 子进程（<code class="code-ref">src/main.zig:~runDaemon</code>），daemon 在 <code class="code-ref">src/daemon.zig</code> 的 <code class="code-ref">Daemon.loop</code> 中以单线程 poll(2) 循环同时监听 listen fd、SIGCHLD self-pipe、所有客户端连接 fd 和 PTY master fd。PTY 分配在 <code class="code-ref">src/pty.zig:spawnInPty</code> 中完成，使用原生 C 绑定（posix_openpt/grantpt/unlockpt），子进程通过 setsid + TIOCSCTTY 获取控制终端，父进程关闭 slave 端。<code class="code-ref">src/window.zig:Window.create</code> 将 PTY master 与 ghostty-vt Terminal 和 TerminalStream 绑定，所有子进程输出经 <code class="code-ref">stream.nextSlice</code> 解析，terminal 状态包含完整屏幕、滚动历史、光标和终端模式。attach 时 <code class="code-ref">repaint</code> 方法（<code class="code-ref">src/window.zig</code>）通过 VT TerminalFormatter 从模拟器状态重新生成字节流，经 <code class="code-ref">altscreen.Filter</code> 过滤掉 alternate-screen 切换序列后发送给客户端。客户端 <code class="code-ref">src/client.zig:attach</code> 将 TTY 切到 raw mode 和 alternate screen，用 poll 三路复用（tty stdin、socket、signal pipe），信号通过 self-pipe trick 处理 SIGWINCH/SIGTERM/SIGHUP。</p>
<p class="audit-callout audit-callout--highlight"><code class="code-ref">src/window.zig:feedDiscarded</code> 与 <code class="code-ref">src/altscreen.zig:Filter</code> 的配合——passthrough 模式下 alt-screen 切换被过滤，被丢弃的字节仍通过 <code class="code-ref">feedDiscarded</code> 注入 terminal 模拟器，且此时 <code class="code-ref">feeding_discarded=true</code> 强制 daemon 端回答终端查询（DSR、DA 等），避免应用程序因查询无响应而挂起（<code class="code-ref">src/window.zig:effectWritePty</code> 的双条件判断）。</p>
<p class="audit-callout audit-callout--highlight"><code class="code-ref">src/keys.zig:Parser</code> 同时支持传统 0x01 escape 字节和 kitty keyboard CSI-u 编码的 prefix 识别，包含 press/repeat/release 事件区分、修饰键过滤（<code class="code-ref">isModifierKey</code>）、caps lock 忽略、序列跨 feed 分片和缓冲区溢出回放——约 300 行状态机有 18 个 test case 覆盖。</p>
<p class="audit-callout audit-callout--doubt">daemon 单线程 poll 模型下，<code class="code-ref">src/daemon.zig:serviceWindow</code> 中对每个 PTY 读取块执行 <code class="code-ref">alt_filter.feed</code> + <code class="code-ref">win.feed</code> + <code class="code-ref">win.feedDiscarded</code>，如果 child 输出极快且 alt-screen 频繁切换，repaint 会产生大量 VT 字节在同一个 poll dispatch 内串行发送，可能造成客户端积压；未见背压或流控机制。</p>
<p class="audit-callout audit-callout--doubt"><code class="code-ref">src/daemon.zig:handleCommand</code> 中 <code class="code-ref">peek</code> 命令将完整屏幕 dump 截断到 <code class="code-ref">protocol.max_payload</code>（1MB），但 <code class="code-ref">src/protocol.zig:max_payload</code> 是 hard limit，对于 512K 行 scrollback 场景可能丢失内容且无截断警告返回给调用方。</p>
<p>agent 自动化场景可直接投入使用 <code class="code-ref">boo new -d -- bash</code> + <code class="code-ref">send/peek/wait</code> 循环，但需注意 <code class="code-ref">wait --text</code> 是客户端轮询（<code class="code-ref">src/main.zig:cmdWait</code> 每 50ms 发一次 peek 控制命令），高频场景下 daemon 负载线性增长；建议在 daemon 端实现 server-side wait 语义以降低 round-trip。</p>
</div>
</section>
